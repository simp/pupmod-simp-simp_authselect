# AGENTS.md

This file provides guidance to AI agents when working with code in this repository.

## What this module does

`simp-simp_authselect` is an early-stage SIMP Puppet module (version `0.1.0`)
that wraps the community `jcpunk/authselect` module and wires it up so that the
`authselect` PAM profiles it generates are **compatible with the SIMP `pam`
module**. On modern Enterprise Linux (EL8+), `authselect` is the supported way
to manage the stack of `*-auth` PAM include files; this module builds a
**custom authselect profile** whose per-section files simply `include` the
matching SIMP-managed PAM fragments, so that `simp/pam` remains the source of
truth for the actual PAM content while `authselect` owns the profile mechanics.

The module is intentionally conservative: **by default it does nothing.** The
whole class is gated behind an on/off toggle (`use_authselect`) that defaults to
`false` via the `simp_options::authselect` seam. When the toggle is off the
class only emits a `notify` explaining that it is disabled
(`manifests/init.pp`).

### Business logic

The module is a single public class; there are no defines and no other classes.

- **`simp_authselect` (`manifests/init.pp`)** — Public entry class (not
  `assert_private()`'d; consumers `include 'simp_authselect'`). Parameters
  (`init.pp`):
  - `$custom_profile_name` (`String`, default `'simp'`) — the name of the custom
    authselect profile this module creates. Files land under
    `custom/${custom_profile_name}` (`init.pp`).
  - `$base_profile` (`Enum['sssd','winbind','nis','minimal']`, default `'sssd'`)
    — the stock authselect base profile the custom profile is derived from
    (`init.pp`).
  - `$authselect_sections` (`Array[String]`, default
    `['fingerprint', 'password', 'smartcard', 'system']`) — the PAM sections to
    generate include files for; each becomes a `<section>-auth` entry and is
    also passed straight through to `pam`'s `auth_sections` parameter
    (`init.pp`, `init.pp`).
  - `$use_authselect` (`Boolean`) — the master switch. Defaults to
    `simplib::lookup('simp_options::authselect', { 'default_value' => false })`
    (`init.pp`), i.e. **off** unless `simp_options::authselect` is set.

  Control flow and resources (`init.pp`):
  - **`if $use_authselect`** guards the entire body. When false, the `else`
    branch is a single `notify { 'Authselect is not enabled, doing nothing': }`
    (`init.pp`) — no `pam`, no `authselect`.
  - **`class { 'pam' }`** (`init.pp`) is declared with
    `auth_sections => $authselect_sections`, handing the section list to the
    SIMP `pam` module so it manages the underlying PAM fragments.
  - **`$contents` reduction** (`init.pp`): folds `$authselect_sections`
    into a hash keyed by `"${section}-auth"`, each value a hash
    `{ 'content' => "include '${custom_profile_name}/${section}-auth'" }`. This
    is the glue — every generated profile file just `include`s the SIMP-managed
    fragment of the same name.
  - **`class { 'authselect' }`** (`init.pp`) is declared with
    `profile_manage => true`, `profile => "custom/${custom_profile_name}"`, and
    a `custom_profiles` hash mapping `$custom_profile_name` to
    `{ base_profile => $base_profile, contents => $contents }`. This is where the
    custom profile is actually built and selected.

### Gotchas / non-obvious details

- **The module is off by default and does nothing.** With no configuration,
  applying `simp_authselect` only produces a `notify` (`init.pp`). To
  activate it, set `simp_options::authselect: true` in Hiera (or pass
  `use_authselect => true`).
- **`$authselect_sections` drives two things at once.** The same array is both
  the `pam` module's `auth_sections` and the set of profile files generated by
  the `$contents` reduction (`init.pp`). Changing it changes both
  what `pam` manages and which include files the profile references — keep them
  in step.
- **This module owns the *wiring*, not the PAM content.** The generated profile
  files contain only `include '<profile>/<section>-auth'` lines
  (`init.pp`); the real PAM content comes from `simp/pam`. Treat `pam` as
  the source of truth for auth behavior.
- **`base_profile` is a closed enum.** Only `sssd`, `winbind`, `nis`, and
  `minimal` are accepted (`init.pp`); anything else fails compilation.
- **Docstrings are still scaffolding.** The class `@summary` is a literal
  `TODO:` placeholder (`init.pp`) which propagates into `REFERENCE.md`. If you
  touch the class, this is worth fixing and regenerating.
- **`simp/oddjob` and `simp/compliance_markup` are declared dependencies but are
  not referenced in `manifests/init.pp`.** They are pulled in as part of the
  SIMP ecosystem (oddjob for authselect's privilege bridge, compliance_markup
  for the SIMP compliance data hierarchy) rather than being `include`d directly
  by this class — do not assume a manifest call site for them.

## The `simp_options` / `simplib::lookup` seam

This is the module's real business-logic seam (the natural target for a
lookup-path unit test). The single call is in `manifests/init.pp`:

| File | Key | `default_value` |
|------|-----|-----------------|
| `init.pp` | `simp_options::authselect` | `false` |

`simp_options::authselect` is the on/off toggle gating the entire module. Keep
routing SIMP feature toggles through `simplib::lookup('simp_options::*', {
'default_value' => ... })` with an explicit default rather than assuming
`simp_options` is included.

## Dependencies

Module dependencies (from `metadata.json`) — all seven are hard dependencies;
there are **no** optional dependencies (no `simplib::assert_optional_dependency`
calls anywhere in the module):

- `simp/pam` `>= 7.0.0 < 8.0.0` (the `pam` class this module wraps)
- `simp/compliance_markup` `>= 3.8.0 < 4.0.0`
- `jcpunk/authselect` `>= 1.0.0 < 2.0.0` (the `authselect` class and its
  `custom_profiles` mechanism)
- `simp/simp_options` `>= 1.0.0 < 2.0.0` (the `simp_options::authselect` toggle)
- `simp/oddjob` `>= 2.0.0 < 3.0.0`
- `simp/simplib` `>= 3.15.0 < 5.0.0` (provides `simplib::lookup`)
- `puppetlabs/stdlib` `>= 6.6.0 < 10.0.0`

Fixture dependencies (from `.fixtures.yml`, checked out for test compilation):
all seven runtime deps above, plus the module itself symlinked in.

Runtime requirement (from `metadata.json` `requirements`): `puppet
>= 7.0.0 < 9.0.0`. Note this module names **`puppet`** and uses the older
`>= 7.0.0` baseline. (SIMP is migrating Puppet → OpenVox; if `metadata.json`
switches this to `openvox`, update this line to match.)

Supported OS matrix (from `metadata.json`): CentOS 9; RedHat 8/9;
OracleLinux 8/9; Rocky 8/9; AlmaLinux 8/9. (Note: CentOS lists only 9, not 8.)

## Repository layout

- `manifests/init.pp` — the sole manifest; the `simp_authselect` class (all
  logic).
- `data/common.yaml` — module data (currently only a commented placeholder).
- `hiera.yaml` — module data hierarchy (v5): OS name+major → OS name → common.
- `metadata.json` — deps, OS matrix, Puppet requirement.
- `spec/classes/init_spec.rb` — rspec-puppet unit tests: exercises the default
  section list, a single-section list, and the disabled (`authselect => false`)
  path, asserting the generated `/etc/authselect/custom/simp/*-auth` files and
  the presence/absence of the `pam` and `authselect` classes.
- `spec/acceptance/suites/default/00_default_spec.rb` — beaker acceptance suite;
  nodesets under `spec/acceptance/nodesets/` (`default.yml`,
  `oel-combined-x64.yml`). **These nodesets are not wired into CI** (see below).
- `REFERENCE.md` — generated Puppet Strings reference (still carries the
  `TODO:` summary).
- `types/`, `templates/`, `files/`, `lib/` — present but **empty** (only the
  skeletal `lib/puppet/{functions,parser/functions,provider,type}` dirs). This
  module has no custom data types, templates, files, or Ruby
  types/providers/functions/facts. Everything it uses comes from the
  dependencies above.
- **No acceptance job in CI.** `.github/workflows/pr_tests.yml` defines six jobs
  only: `puppet-syntax`, `puppet-style`, `ruby-style`, `file-checks`,
  `releng-checks`, and `spec-tests` (`spec-tests` runs `bundle exec rake spec`
  on a Puppet 7.x/8.x matrix). There is no beaker/nodeset/`BEAKER_HYPERVISOR`
  step — the shipped nodesets are run manually, not in CI.

## Conventions

- Fix the `@summary` / `@param` puppet-strings docstrings on the class if you
  edit it — the `@summary` is still a `TODO:` placeholder and drives
  `REFERENCE.md`. Regenerate `REFERENCE.md` after changing docs or parameters.
- Keep this module as *wiring only*: generate authselect profile files that
  `include` the SIMP-managed PAM fragments, and let `simp/pam` own the actual
  PAM content. Don't inline PAM logic here.
- Keep `$authselect_sections` as the single source for both the `pam`
  `auth_sections` and the generated `$contents` include files.
- Continue routing SIMP feature toggles through
  `simplib::lookup('simp_options::*', { 'default_value' => ... })` rather than
  assuming `simp_options` is included.
- Several baseline files carry a **puppetsync** notice — e.g. `Gemfile`, `spec/spec_helper.rb`, `.github/workflows/pr_tests.yml`, and the `.gitignore`/`.pdkignore` dotfiles — so they are baseline-managed and the next sync overwrites local edits. Check each file's header for the notice rather than treating this list as exhaustive; push changes to any such file upstream to the baseline, not here.
- Match the existing 2-space Puppet indentation and aligned-arrow parameter
  style used in `manifests/init.pp`.
